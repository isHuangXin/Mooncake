# Mooncake KVCache 分级存储架构详解

## 问题

> Mooncake是怎么做KVCache的分级存储的，哪些数据放在GPU HBM，哪些数据放在CPU DRAM，哪些数据放在SSD。这里的SSD是指单节点的SSD吗？例如一个8卡GPU的节点，还是8个8卡GPU节点构成的多节点SSD池？

---

## 回答

### 一、整体存储层级架构

根据架构图和代码分析，Mooncake 的 KVCache 存储分为 **3 个主要层级**：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        Mooncake 存储层级架构                                  │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │  L1: GPU VRAM (HBM)                                                 │    │
│  │  • 位置：每个 Inference Instance 私有                                 │    │
│  │  • 管理者：SGLang/vLLM 的 Paged KVCache                              │     │
│  │  • 特点：最快访问，容量有限（~80GB/卡）                                  │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                               ↕ GPU↔CPU 传输                                 │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │  L2: Host DRAM                                                      │    │
│  │  • 位置：每个 Inference Instance 私有 (单节点)                         │    │
│  │  • 管理者：HiCache Host Memory Pool                                  │    │
│  │  • 特点：中等速度，容量较大（~1TB/节点）                                 │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                               ↕ RDMA / TCP                                  │
│  ┌─────────────────────────────────────────────────────────────────────┐    │
│  │  L3: Mooncake Store (分布式)                                         │    │
│  │  ┌──────────────────┐  ┌──────────────────┐  ┌──────────────────┐   │    │
│  │  │ 节点1 DRAM Pool   │  │ 节点2 DRAM Pool   │  │ 节点N DRAM Pool  │   │    │
│  │  │ (Segment 1,2...) │  │ (Segment 3,4...) │  │ (Segment N...)   │   │    │
│  │  └──────────────────┘  └──────────────────┘  └──────────────────┘   │    │
│  │           └───────────────────┼───────────────────┘                 │    │
│  │                               ↕ 异步持久化                            │    │
│  │  ┌─────────────────────────────────────────────────────────────┐    │    │
│  │  │  分布式文件系统 (DFS/3FS) - 多节点共享                          │    │    │
│  │  │  • 跨节点挂载的共享存储 (如 /mnt/3fs)                           │    │    │
│  │  │  • 用于持久化备份和冷数据存储                                   │    │    │
│  │  └─────────────────────────────────────────────────────────────┘    │    │
│  └─────────────────────────────────────────────────────────────────────┘    │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 二、各层存储的数据放置策略

#### 2.1 GPU HBM (L1) - 热数据

**数据特点**：
- 正在进行 Prefill/Decode 计算的 KVCache
- 最近访问、命中率最高的 KVCache

**管理机制**：
- 由推理引擎（SGLang/vLLM）的 **Paged KVCache** 管理
- 使用 **RadixTree** 组织前缀共享
- 当空间不足时，驱逐到 L2

**代码参考** (hicache-design.md):
```
L1 和 L2 KV caches 是每个推理实例私有的，
而 L3 KV cache 在集群内的所有推理实例之间共享。
```

#### 2.2 Host DRAM (L2) - 温数据

**数据特点**：
- 从 L1 驱逐但近期可能重用的 KVCache
- 从 L3 预取等待加载到 L1 的数据

**管理机制**：
- 由 **HiCache Host Memory Pool** 管理
- 每个 Inference Instance 私有（单节点）
- 支持异步 Write-back 到 L3

#### 2.3 Mooncake Store (L3) - 分布式冷数据池

L3 是最复杂的层级，实际上包含 **两个子层**：

##### 2.3.1 分布式 DRAM Pool（主要存储）

**数据特点**：
- 全局共享的 KVCache（如系统提示、常见前缀）
- 跨实例复用的数据

**架构**：
```
┌────────────────────────────────────────────────────────────────┐
│                  Distributed KVCache Pool                      │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐          │
│  │   节点1       │  │   节点2       │ │   节点N       │          │
│  │ CPU/DRAM/SSD │  │ CPU/DRAM/SSD │  │ CPU/DRAM/SSD │          │
│  │  (Segment)   │  │  (Segment)   │  │  (Segment)   │          │
│  └──────────────┘  └──────────────┘  └──────────────┘          │
│           ↑              ↑              ↑                      │
│           └──────────────┼──────────────┘                      │
│                    RDMA 互联                                    │
└────────────────────────────────────────────────────────────────┘
```

##### 2.3.2 分布式文件系统 (DFS/3FS) - 持久化层

**数据特点**：
- 从 DRAM Pool 异步备份的数据
- 长期未访问的冷数据
- 需要持久化的重要数据

**关键配置** (mooncake-store.md):
```bash
# 启动 master 时指定 DFS 挂载点
./mooncake_master --root_fs_dir=/path/to/dfs_mount_point

# DFS 必须在所有客户端节点上挂载到相同路径
```

### 三、关于 SSD 的问题：单节点 vs 多节点？

#### 回答：**是多节点共享的分布式文件系统 (DFS)**

根据代码和文档分析：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                          SSD 存储架构                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ❌ 不是单节点 SSD：                                                          │
│     • Mooncake Store 的 SSD 层不是直接访问本地 NVMe SSD                        │
│     • 而是通过 DFS（分布式文件系统）抽象                                         │
│                                                                             │
│  ✅ 是多节点共享 DFS：                                                         │
│     • 要求：root_fs_dir 必须是所有节点都能访问的共享挂载点                         │
│     • 例如：3FS、NFS、Lustre 等分布式文件系统                                    │
│     • 所有节点挂载到相同路径（如 /mnt/3fs）                                      │
│                                                                             │
│  架构示意：                                                                   │
│                                                                             │
│  ┌───────────────┐  ┌───────────────┐  ┌───────────────┐                    │
│  │   GPU 节点 1   │  │   GPU 节点 2   │  │   GPU 节点 N   │                   │
│  │  /mnt/3fs ────┼──┼── /mnt/3fs ───┼──┼── /mnt/3fs    │                    │
│  └───────────────┘  └───────────────┘  └───────────────┘                    │
│           │                │                 │                              │
│           └────────────────┼─────────────────┘                              │
│                            ↓                                                │
│              ┌─────────────────────────────┐                                │
│              │    分布式文件系统后端          │                                 │
│              │  (3FS / NFS / Lustre 等)    │                                │
│              │                             │                                │
│              │  ┌─────┐ ┌─────┐ ┌─────┐    │                                 │
│              │  │SSD 1│ │SSD 2│ │SSD N│    │  ← 物理 SSD 分布在多节点           │
│              │  └─────┘ └─────┘ └─────┘    │                                 │
│              └─────────────────────────────┘                                │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

**关键代码证据** (mooncake-store.md):
```
When the user specifies `--root_fs_dir=/path/to/dir` when starting the master, 
and this path is a valid DFS-mounted directory on **all machines where the clients reside**, 
Mooncake Store's tiered caching functionality will work properly.
```

### 补充：Transfer Engine 支持的 NVMe-oF

Transfer Engine 还支持 **NVMe over Fabric (NVMe-oF)**，可以远程访问其他节点的 NVMe SSD：

```cpp
// nvmeof_transport.h
class NVMeoFTransport : public Transport {
    // 支持通过网络直接访问远程 NVMe SSD
    // 使用 GPUDirect Storage (GDS) 实现 GPU 直接读写远程 SSD
};
```

### 四、数据流动示意

```
                     写入路径 (Write-back)
        ┌─────────────────────────────────────────────┐
        │                                             │
        ▼                                             │
┌─────────────┐    驱逐      ┌─────────────┐    备份     ┌─────────────┐
│   L1: GPU   │ ─────────→  │  L2: Host   │ ─────────→  │ L3: Remote  │
│    HBM      │             │    DRAM     │  (异步)     │ DRAM + DFS  │
└─────────────┘             └─────────────┘             └─────────────┘
        ▲                          ▲                          │
        │                          │                          │
        │       加载                │        预取              │
        └──────────────────────────┴──────────────────────────┘
                     读取路径 (Prefetch)
```

**数据流动规则**：

| 操作 | 触发条件 | 数据流向 |
|------|----------|----------|
| L1→L2 | GPU 内存不足驱逐 | 同步写入本地 Host DRAM |
| L2→L3 DRAM | Write-back 策略触发 | RDMA 写入远程节点 DRAM |
| L3 DRAM→L3 DFS | Put 操作时异步 | 文件系统写入 |
| L3 DFS→L2 | Get 时 DRAM 未命中 | 文件系统读取到本地 DRAM |
| L3 DRAM→L2 | Prefetch 预取 | RDMA 读取到本地 DRAM |
| L2→L1 | 计算需要 | cudaMemcpy / GPU Kernel |

### 五、总结

| 存储层级 | 物理位置 | 共享范围 | 主要用途 |
|---------|----------|----------|----------|
| **L1 GPU HBM** | 每张 GPU 卡 | Instance 私有 | 正在计算的热数据 |
| **L2 Host DRAM** | 单节点 CPU 内存 | Instance 私有 | 本地温数据缓存 |
| **L3 Remote DRAM** | **多节点** 分布式 | **全局共享** | 跨实例共享数据 |
| **L3 DFS/SSD** | **多节点** 分布式文件系统 | **全局共享** | 持久化冷数据 |

**回答您的问题**：
- Mooncake 的 SSD 层是 **多节点构成的分布式文件系统 (DFS)**
- 不是单个 8 卡节点的本地 SSD
- 而是类似 3FS/NFS 这样的共享存储，所有节点通过相同路径挂载访问
- 这使得 KVCache 可以在整个集群范围内共享和持久化
