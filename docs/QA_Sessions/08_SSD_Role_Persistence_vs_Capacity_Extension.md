# Mooncake 中 SSD 的角色：持久化层 vs 容量扩展

## 问题

> 在 Mooncake 中，SSD 仅仅只是作为 KVCache 的持久化吗？核心的 KVCache 都是存在 HBM 和 DRAM 中，其中持久化层 SSD 存储的 KVCache 与 DRAM 中重复，而不是作为 DRAM 的扩展补充来存放额外的 KVCache？

---

## 回答

### 结论

**是的，你的理解基本正确。** 在 Mooncake 的 HiCache 架构中，SSD（通过 DFS/3FS）**主要作为持久化/备份层**，而不是作为 DRAM 的容量扩展。

## 二、关键证据

### 2.1 官方文档说明

来自 `mooncake-store.md`：

> **Multi-layer storage support​​**: Mooncake Store supports **offloading cached data from RAM to SSD**, further balancing cost and performance to improve storage system efficiency.

关键词是 **offloading**（卸载），而不是 **extending**（扩展）。这意味着：
- SSD 存储的数据是从 RAM **拷贝/备份** 过去的
- 不是 RAM 放不下的数据放到 SSD

### 2.2 HiCache Write-back 策略

来自 `hicache-design.md`：

```
**Configurable Write-back Policies**: 
* **write_through**: Every access is immediately written back to the next level.
* **write_through_selective**: Data is written back only after access frequency exceeds a threshold.
* **write_back**: Data is written back to the next level only when evicted from upper level.
```

这些策略都是关于**何时备份数据到下一层**，而不是**何时将数据移动到下一层以腾出空间**。

### 2.3 代码中的 Offload 机制

```cpp
// storage_backend.h
virtual tl::expected<int64_t, ErrorCode> BatchOffload(
    const std::unordered_map<std::string, std::vector<Slice>>& batch_object,
    std::function<ErrorCode(...)> complete_handler) = 0;
```

函数名为 `BatchOffload`，表示"批量卸载"，即将数据从快速存储**备份**到慢速存储，而非**迁移**。

## 三、SSD 角色的详细分析

### 3.1 两种可能的设计

| 设计模式 | 说明 | Mooncake 是否采用 |
|---------|------|------------------|
| **容量扩展** | DRAM 放不下的冷数据移动到 SSD，SSD 存储独占数据 | ❌ 否 |
| **持久化备份** | SSD 存储 DRAM 数据的副本，用于故障恢复和跨实例共享 | ✅ 是 |

### 3.2 为什么选择持久化备份而非容量扩展？

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     设计考量                                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   1. LLM 推理的数据访问特点                                                  │
│      • KVCache 访问是突发的、不可预测的                                      │
│      • 一旦需要某个 prefix 的 KVCache，需要立即全部加载                       │
│      • SSD 延迟（~10μs）对于 prefill 阶段是不可接受的                        │
│                                                                             │
│   2. 分布式 DRAM 提供足够容量                                                │
│      • 100 节点 × 500GB DRAM = 50TB 分布式 KVCache 池                       │
│      • 对于大多数场景，分布式 DRAM 容量已经足够                              │
│      • 不需要 SSD 来扩展容量                                                │
│                                                                             │
│   3. SSD 作为备份的价值                                                      │
│      • 持久化：节点重启后数据不丢失                                          │
│      • 共享：新加入的节点可以从 SSD 加载历史数据                             │
│      • 成本：长期存储使用 SSD 比 DRAM 便宜                                   │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.3 数据流示意

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     数据流向                                                 │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   写入路径 (Write-back):                                                    │
│   ┌───────────┐      ┌───────────┐      ┌───────────┐      ┌───────────┐   │
│   │  GPU HBM  │ ───► │ Host DRAM │ ───► │ Remote    │ ───► │ DFS/SSD   │   │
│   │  (计算)   │      │ (L2 缓存)  │      │ DRAM      │      │ (备份)    │   │
│   └───────────┘      └───────────┘      └───────────┘      └───────────┘   │
│                                               │                   │         │
│                                               │                   │         │
│                                               └───────────────────┘         │
│                                                  两者数据相同！              │
│                                                  SSD 是 DRAM 的备份         │
│                                                                             │
│   读取路径 (Prefetch):                                                      │
│   ┌───────────┐      ┌───────────┐      ┌───────────┐                      │
│   │  GPU HBM  │ ◄─── │ Host DRAM │ ◄─── │ Remote    │  ★ 主要数据源         │
│   │  (计算)   │      │ (L2 缓存)  │      │ DRAM      │                      │
│   └───────────┘      └───────────┘      └───────────┘                      │
│                                                                             │
│   注意：Prefetch 主要从 Remote DRAM 读取，而非 SSD                           │
│   SSD 只在 DRAM 数据丢失时作为恢复源                                         │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 四、与"容量扩展"设计的对比

### 4.1 如果 SSD 是容量扩展（Mooncake 未采用）

```
假设：DRAM 总容量 1TB，需要存储 2TB KVCache

容量扩展模式：
┌─────────────────────────────────────────┐
│  DRAM (1TB)        │  SSD (1TB)         │
│  热数据 A, B, C    │  冷数据 D, E, F    │
│  (独占)            │  (独占)            │
└─────────────────────────────────────────┘
• 总容量: 2TB (DRAM + SSD)
• A, B, C 只在 DRAM
• D, E, F 只在 SSD
• 访问 D 需要从 SSD 加载 (慢！)
```

### 4.2 Mooncake 的实际设计（持久化备份）

```
Mooncake 实际设计：
┌─────────────────────────────────────────┐
│  DRAM (1TB)        │  SSD (1TB)         │
│  数据 A, B, C      │  数据 A, B, C 备份  │
│  (主存储)          │  (持久化副本)       │
└─────────────────────────────────────────┘
• 总容量: 1TB (DRAM 决定)
• A, B, C 在 DRAM 和 SSD 都有副本
• 访问始终从 DRAM 读取 (快！)
• SSD 用于：
  - 故障恢复
  - 跨实例共享
  - 长期持久化
```

## 五、总结

| 问题 | 答案 |
|------|------|
| SSD 仅作为持久化层？ | ✅ **是**，主要用于备份和持久化 |
| 核心 KVCache 存在 HBM 和 DRAM？ | ✅ **是**，这是主要工作存储 |
| SSD 数据与 DRAM 重复？ | ✅ **是**，SSD 是 DRAM 数据的副本 |
| SSD 作为 DRAM 的容量扩展？ | ❌ **否**，不是用来存放 DRAM 放不下的数据 |

### 设计理由

1. **性能优先**：LLM 推理对延迟敏感，SSD 的 ~10μs 延迟无法满足 prefill 需求
2. **分布式 DRAM 提供足够容量**：跨节点池化的 DRAM 容量已足够大
3. **SSD 价值在于持久化**：
   - 节点故障恢复
   - 新节点快速预热
   - 长期数据保留
   - 跨实例数据共享

### 如果需要更大容量？

如果确实需要比分布式 DRAM 更大的 KVCache 容量，应该：
1. **增加节点数量**（扩展 DRAM 池）
2. **使用你提出的 Flat Memory 设计**（将 SSD 作为真正的容量扩展层，但需接受更高的 TTFT）
