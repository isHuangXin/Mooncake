# 调度算法 TTFT 对比：数据来源分析

## 问题

> 这里的 TTFT 时间是指 KVCache 主要是从 HBM 和 DRAM 中取，而不是从 SSD 中取来测试的吗？

---

## 回答

### 结论

**是的，你的理解正确。** 图中的 TTFT 测试数据主要反映的是从 **HBM 和 DRAM**（包括本地和远程 DRAM）读取 KVCache 的性能，而不是从 SSD 读取。

### 1.1 为什么确定不是从 SSD 读取？

| 证据 | 说明 |
|------|------|
| **TTFT 量级** | 3-20 秒级别，符合大量 token 的 DRAM→GPU 传输时间 |
| **策略差异** | 缓存感知策略的优势体现在 DRAM 缓存命中，而非 SSD |
| **HiCache 设计** | Prefetch 主要从 Remote DRAM 读取，SSD 是备份层 |
| **延迟特征** | 如果从 SSD 读取，TTFT 会更长且策略间差异模式不同 |

## 三、TTFT 组成分析

### 3.1 TTFT 的主要组成部分

```
TTFT = 调度延迟 + 数据加载延迟 + Prefill 计算延迟

┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   数据加载延迟（主要差异来源）：                                             │
│                                                                             │
│   情况 1: L1 命中 (HBM)                                                     │
│   ┌─────────┐                                                              │
│   │   ~0    │  数据已在 GPU，无需加载                                       │
│   └─────────┘                                                              │
│                                                                             │
│   情况 2: L2 命中 (本地 DRAM)                                               │
│   ┌─────────────────┐                                                      │
│   │   ~10-100ms     │  PCIe 传输: DRAM → GPU                               │
│   └─────────────────┘                                                      │
│                                                                             │
│   情况 3: L3 命中 (远程 DRAM via RDMA)                                      │
│   ┌─────────────────────────────┐                                          │
│   │   ~100ms-1s                 │  RDMA 读取 + PCIe 传输                   │
│   └─────────────────────────────┘                                          │
│                                                                             │
│   情况 4: L3 命中 (SSD/DFS) - 测试中很少发生                                │
│   ┌─────────────────────────────────────────┐                              │
│   │   ~1-10s (取决于数据量)                  │  SSD 读取 + 网络 + PCIe     │
│   └─────────────────────────────────────────┘                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 各策略的缓存命中情况估算

| 策略 | TTFT | 估算缓存命中情况 |
|------|------|-----------------|
| **全局缓存感知** | 3.07s | 高 L1/L2 命中率，少量 L3 DRAM Prefetch |
| **本地缓存感知** | 3.58s | 高本地缓存命中，但可能负载不均 |
| **负载均衡** | 5.27s | 低缓存命中，较多 L3 DRAM Prefetch |
| **随机调度** | 19.65s | 极低缓存命中，大量 L3 DRAM Prefetch |

## 四、为什么不是从 SSD 读取？

### 4.1 HiCache 的 Prefetch 策略

来自 `hicache-design.md`：

> **Prefetch from L3**: Data prefetching is one of HiCache's core optimization techniques, designed to proactively load KV caches from L3 storage into local L2 memory.

L3 存储的主要数据源是 **分布式 DRAM**，不是 SSD：

```
L3 Prefetch 优先级：
1. Remote DRAM (via RDMA)  ← 主要来源，延迟 ~2μs
2. SSD/DFS (备份)          ← 仅在 DRAM 数据不存在时使用
```

### 4.2 如果从 SSD 读取，TTFT 会是什么样？

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                     假设从 SSD 读取的 TTFT 估算                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   假设：                                                                    │
│   • 需要加载 10GB KVCache                                                   │
│   • SSD 带宽: ~7 GB/s (NVMe)                                               │
│   • 网络带宽: ~25 GB/s (200Gbps RDMA)                                      │
│                                                                             │
│   从 DRAM (RDMA):                                                          │
│   • 传输时间: 10GB / 25GB/s ≈ 0.4s                                         │
│   • 加上调度和计算: ~3-5s (符合图中数据)                                    │
│                                                                             │
│   从 SSD:                                                                   │
│   • SSD 读取: 10GB / 7GB/s ≈ 1.4s                                          │
│   • 网络传输: + 0.4s                                                       │
│   • 总计: ~5-10s (比图中数据更长)                                          │
│                                                                             │
│   如果大量请求都从 SSD 读取：                                               │
│   • 随机调度的 TTFT 可能达到 30-60s，而非 19.65s                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

## 五、总结

| 问题 | 答案 |
|------|------|
| TTFT 测试的数据来源？ | 主要是 **HBM + DRAM**（本地和远程） |
| 是否涉及 SSD？ | **很少**，SSD 主要作为备份，Prefetch 优先从 DRAM 读取 |
| 策略差异的原因？ | 缓存命中率不同导致 DRAM Prefetch 量不同 |

### 关键洞察

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                                                                             │
│   全局缓存感知 vs 随机调度 的 TTFT 差异 (3.07s vs 19.65s, 6.4倍)           │
│                                                                             │
│   差异来源：                                                                │
│   • 全局缓存感知：高缓存命中 → 少量 RDMA Prefetch → 低 TTFT                │
│   • 随机调度：低缓存命中 → 大量 RDMA Prefetch → 高 TTFT                    │
│                                                                             │
│   如果是从 SSD 读取，差异会更大（可能 10-20 倍）                            │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

图中的 TTFT 数据符合 **从 DRAM 读取** 的性能特征，而非从 SSD 读取。
